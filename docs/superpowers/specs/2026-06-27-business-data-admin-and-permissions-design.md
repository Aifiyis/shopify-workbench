# 业务数据后台、员工职位与权限设计

## 目标

为 `sku_match_product_type`、`product_processing_craft` 和工艺层级增加可查询、可维护的中文后台页面，同时建立适合公司内部系统的员工、职位和权限模型。

系统只服务一套全局业务数据，不按公司名称隔离。员工资料中的公司名称仅作为普通信息字段。所有登录用户可以查看业务数据，修改能力由账号角色权限和员工职位权限共同决定。

## 数据模型

### 产品类型主数据

新增 `product_types` 作为中文名称唯一来源：

- `id`
- `chinese_name`，唯一
- `deleted_at`
- timestamps

`sku_match_product_type` 和 `product_processing_craft` 分别增加 `product_type_id`。现有两表中文名称取并集回填主表，不为没有 SKU 的订单处理配置制造空 SKU 记录。

为降低 SQLite 表重建风险，现有 `chinese_name` 文本列本期保留为历史快照；新页面和新业务逻辑以 `product_type_id` 关联为准。产品类型名称发生变化时，由应用服务同步历史快照。冗余列清理留给后续独立迁移。

### SKU 产品类型

`sku_match_product_type` 增加：

- `product_type_id`
- `product_lister_employee_id`，可空
- `deleted_at`

现有 `product_lister` 文本保留为历史快照。新增和修改记录时，上品人使用员工 ID，候选人员必须拥有广告或运营职位。

### 订单处理配置

`product_processing_craft` 增加：

- `product_type_id`，一项产品类型最多一条有效配置
- `order_processor_employee_id`
- `artwork_processor_employee_id`
- `procurement_processor_employee_id`
- `settlement_method`
- `deleted_at`

三个处理人均为单选。候选人员分别来自订单处理、图画处理和采购职位。现有处理人文本列保留为历史快照；采购处理人括号中的“月结”“周结”等内容拆入 `settlement_method`。

### 工艺层级

数据库表继续使用 `processing_craft_nodes`，代码模型继续使用 `ProcessingCraftNode`，但所有界面文案统一使用：

- 工艺层级管理
- 工艺项
- 工艺名称
- 上级工艺
- 工艺路径
- 新建工艺
- 管理工艺

表增加 `deleted_at`。有有效下级工艺或被订单处理配置引用时禁止删除。

### 员工、职位和权限

新增 `employees`：

- `id`
- `name`
- `company_name`，普通可搜索文本，不用于数据隔离
- `supervisor_id`，可空，自关联直属上级
- `admin_id`，可空且唯一，关联登录账号
- `is_active`
- `deleted_at`
- timestamps

新增 `positions`：

- `id`
- `name`
- `code`，唯一稳定标识
- `is_active`
- `deleted_at`
- timestamps

新增 `employee_position` 多对多关系，允许一名员工拥有多个职位。

新增 `permissions`：

- `id`
- `name`
- `code`，唯一
- `is_delegable`，是否允许下放
- timestamps

新增 `position_permission` 和 `role_permission`。账号有效权限为角色权限与关联员工职位权限的并集；超级管理员拥有全部权限。

首批职位：广告、运营、采购、订单处理、图画处理。首批管理权限至少包括：

- `sku_product_types.manage`
- `order_processing.manage`
- `processing_crafts.manage`
- `admin_accounts.manage`
- `employees.manage`
- `positions.manage`
- `permissions.assign`

广告和运营职位默认获得 SKU 产品类型维护权限。采购、订单处理和图画处理职位默认获得订单处理配置与工艺层级维护权限。

## 权限规则

- 所有已登录账号可以查看和查询 SKU、产品类型、订单处理配置和工艺层级。
- 超级管理员拥有全部操作权限。
- 管理员角色默认拥有业务维护、非超级管理员账号管理、员工管理、职位管理和可下放权限分配能力。
- 普通员工通过职位权限获得新增、修改和软删除能力。
- `admin_accounts.manage` 独立于员工管理权限，未来可授予人力职位。
- 只有超级管理员可以创建、修改或删除超级管理员和管理员角色账号。
- 管理员或获得账号管理权限的普通员工只能管理员工角色账号。
- 允许管理员跨公司名称管理员工和账号，公司名称不构成权限边界。
- 管理员可以维护职位及其权限，但只能授予自己已拥有且 `is_delegable = true` 的权限。
- 员工软删除或停用后不再进入新记录下拉框，历史业务记录继续显示关联姓名或历史文本。

管理员账号改为软删除。认证查询自动排除已删除账号。

## 历史数据初始化

从现有业务字段提取员工姓名并建立员工档案：

- 上品人默认分配广告职位，后续可人工调整为运营或多职位。
- 订单处理人分配订单处理职位。
- 图画处理人分配图画处理职位。
- 采购处理人去除结算括号后分配采购职位。
- 同名员工合并职位，不创建重复档案。
- `李鑫/万芸君`、`预览图` 等无法映射到单选人员的值保留为未关联历史文本，不自动猜测。

创建 `qxadmin` 管理员账号：邮箱为 `test@qq.com`，上级为当前超级管理员 `Admin`，同时创建关联员工档案。实施时使用用户提供的密码并立即哈希，明文密码不写入源码、日志或设计文档。

## 页面设计

### 全局导航

统一 Blade 后台布局和中文导航：

- 工作台
- 数据处理
- SKU 产品类型
- 订单处理配置
- 工艺层级管理
- 员工与职位
- 管理员管理

页面采用紧凑表格、服务端搜索、筛选和分页，适合重复维护工作。不使用营销式布局或大卡片堆叠。

### SKU 产品类型

页面包含两个页签：

1. `SKU 映射`：搜索 original SKU、cleaned SKU、产品类型和上品人；支持新增、修改和软删除。
2. `产品类型`：维护统一中文名称，显示 SKU 数量及订单处理配置状态。

SKU 表单的产品类型为可搜索、可创建的自动补全控件。新名称通过 AJAX 先写入 `product_types`，再自动选中。上品人为按职位过滤的可搜索单选。

有关联 SKU 或订单处理配置时禁止删除产品类型。

### 订单处理配置

产品类型只能从 `product_types` 选择，不能在此页面创建任意名称。

工艺选择使用可搜索、按层级缩进展示的下拉框，已选值显示完整工艺路径。旁边提供：

- `新建工艺`：打开轻量弹窗，通过 AJAX 创建；成功后不刷新页面，直接加入下拉框并自动选中。
- `管理工艺`：进入独立工艺层级管理页，并保留可返回订单处理配置的来源参数。

订单、图画和采购处理人使用按职位过滤的可搜索单选。结算方式为可搜索、可创建下拉框，默认提供月结和周结。

### 工艺层级管理

使用独立页面展示工艺名称、完整路径、层级关系和引用状态，支持搜索、新增、修改和软删除。层级深度不设业务上限。

### 员工与职位

页面包含员工档案和职位权限两个页签：

- 员工档案：姓名、公司名称、直属上级、可选账号、多个职位、状态。
- 职位权限：职位名称、启用状态和可授予权限。

经理可跨公司名称管理员工；公司名称输入框提供历史值自动补全，但允许新值。

### 删除交互

所有删除按钮打开中文确认弹窗，确认后执行软删除。本期不提供回收站、恢复和永久删除入口。

## 中文化

Laravel 默认语言改为 `zh_CN`，建立中文验证、分页和认证语言文件。以下范围全部中文化：

- 登录页
- 工作台
- 管理员管理
- 订单与报表页面
- 数据处理页面
- 新增业务管理页面
- 页面标题、导航、字段、按钮、状态、空数据提示、验证提示和确认弹窗

代码字段名、路由名和数据库字段继续使用英文。

## 控制器与前端边界

每个业务资源使用独立控制器、表单请求和 Policy。列表查询使用服务端分页，不把全部 SKU 发送到浏览器。

可搜索选择框使用一个支持搜索、创建选项和自定义层级渲染的轻量组件。快速创建产品类型和工艺使用 JSON 端点、CSRF 校验和 Policy 授权；成功响应返回新记录 ID、显示名称和完整路径。

删除、快速创建和权限错误都返回明确中文消息。AJAX 失败不关闭弹窗，并保留用户输入。

## 测试与上线安全

- 使用 SQLite `:memory:` 运行定向 Feature/Unit 测试，不执行 `migrate:fresh` 触碰本地数据库。
- 测试产品类型回填、员工姓名归一化、采购结算拆分及无法识别文本保留。
- 测试列表查询、分页、增改、软删除和依赖删除保护。
- 测试角色权限、职位权限、可下放权限限制和账号角色上限。
- 测试 AJAX 新建产品类型、新建工艺及下拉框返回数据。
- 测试 `qxadmin` 使用哈希密码且可正常登录。
- 实施迁移前备份 `database/database.sqlite`，迁移后核对现有 SKU、订单处理和工艺数量。
- 使用 Playwright 验证桌面和移动视口的中文导航、表格、表单、弹窗和下拉框，无文字遮挡或横向破版。

## 非目标

- 不做多租户或按公司隔离业务数据。
- 不提供软删除回收站或恢复页面。
- 不在本期删除现有中文名称和处理人历史文本列。
- 不把职位名称直接写入 `admins` 单列。
